Section 01

System Overview & Design Principles

The AI Accounting Migration System is a production-grade pipeline that automates migration of financial data from Reckon Desktop to Reckon One. It is built on a layered architecture that prioritises determinism, cost efficiency, and human oversight.

Design Principles

PrincipleImplementation
Rules-firstDeterministic Reckon rules applied before any LLM call. 60–80% of accounts resolved at zero AI cost.
Dual-model validationPrimary + validator LLM run in parallel. Agreement required for auto-approval. Prevents hallucinations.
Human-in-the-loopLow-confidence and disagreed mappings surfaced to accountant via UI. No unreviewed data reaches import.
AutoLearn memoryEvery approved mapping persisted to SQLite + RAG. Next migration reuses without LLM or human intervention.
Async concurrencyAll agent calls use asyncio + Semaphore(2). Dual-model calls run in parallel via run_in_executor.
Stateful pipelineLangGraph state machine with MemorySaver checkpoint. Full state recoverable across interruptions.

Technology Stack

Section 03

LangGraph State Machine

The pipeline is implemented as a directed acyclic graph (DAG) using LangGraph's StateGraph. State is typed, accumulated across nodes, and checkpointed with MemorySaver.

State Schema (MigrationState)

class MigrationState(TypedDict):
    # ── Inputs ─────────────────────────────────────────────────
    source_accounts  : list[dict]      # Reckon Desktop COA records
    target_accounts  : list[dict]      # Reckon One target accounts
    journal_entries  : list[dict]      # General journal entries
    source_tb        : dict            # {code: (name, amount)}
    migrated_tb      : dict            # {code: amount}
    cutoff_date      : str             # "YYYY-MM-DD"

    # ── Outputs — accumulated with operator.add ─────────────────
    mappings         : Annotated[list, operator.add]
    anomalies        : Annotated[list, operator.add]
    recon_result     : dict
    hitl_queue       : Annotated[list, operator.add]
    errors           : Annotated[list, operator.add]
    run_complete     : bool
    progress_log     : Annotated[list, operator.add]

Node Definitions & Routing

hitlhitl_node()Pass-through (UI handles)→ anomaly always

# Graph construction — graph.py
g = StateGraph(MigrationState)
g.set_entry_point("mapping")
g.add_conditional_edges("mapping", route_after_mapping,
                        {"hitl": "hitl", "anomaly": "anomaly"})
g.add_edge("hitl", "anomaly")
g.add_edge("anomaly", "reconcile")
g.add_edge("reconcile", "report")
g.add_edge("report", END)
graph = g.compile(checkpointer=MemorySaver())

Section 05

AI Agents — agents.py

Three async agents implement the AI reasoning layer. All call the configured LLM API with exponential backoff retry on rate limits.

mapping_agent_async()

Processes each source account through a 4-stage resolution pipeline:

1
Direct type match via COA_TYPE_MAP
If source type key exists in rules map → find first target with matching Reckon One type. Confidence: 95%. Source: rules. No LLM called.
2
Memory lookup via lookup_mapping()
SQLite query by (name, type) composite key. If found → reuse previously approved mapping. Source: memory. No LLM called.
3
Dual-model LLM inference
Primary (Llama 3.3 70B Versatile) + Validator (Llama 3.1 8B Instant) run in parallel via asyncio.gather(). Both receive: Reckon context + RAG few-shot examples. Average confidence computed. Source: llm.
4
Auto-approve or HITL queue
avg_conf ≥ AUTO_THRESHOLD and models agree → APPROVED. avg_conf ≥ 60 → REVIEW. avg_conf < HITL_THRESHOLD or disagreement → HITL QUEUE.
# Semaphore limits concurrent LLM API calls to avoid rate limits
_sem = asyncio.Semaphore(2)

async with _sem:
    primary_task   = loop.run_in_executor(None, _call, MODEL_PRIMARY,   system, user, 400)
    validator_task = loop.run_in_executor(None, _call, MODEL_VALIDATE,  system, user, 300)
    primary, validator = await asyncio.gather(primary_task, validator_task)

agree    = (primary.get("target_code") == validator.get("target_code"))
avg_conf = (int(primary["confidence"]) + int(validator["confidence"])) // 2

Section 06

Memory & RAG Store

Two complementary persistence mechanisms provide the system's learning capability.

SQLite Memory (memory.py)

Stores approved mappings by composite key (account_name, account_type). Queried before any LLM call.

# lookup_mapping() — O(1) exact match
SELECT target_code, target_name, confidence, approved_by
FROM mappings
WHERE account_name = ? AND account_type = ?
ORDER BY confidence DESC LIMIT 1

RAG Store (rag_store.py) — In-Process Vector Search

The system uses a custom, dependency-free RAG implementation. There is no external vector database (no ChromaDB, no Qdrant, no Pinecone). Embeddings are 64-dimensional character-level hash vectors stored as BLOBs in the same SQLite file. Similarity search is performed with NumPy cosine dot product entirely in-process.

AspectCurrent implementation
Embedding modelCustom 64-dim character-level hash (no ML model, no API call)
Vector storageSQLite rag_entries table — BLOB column (float64 numpy array)
Similarity metricCosine similarity via numpy.dot() — computed in-process
External vector DBNone — no ChromaDB, Qdrant, Pinecone, or Weaviate dependency
RetrievalTop-3 nearest neighbours returned as few-shot examples injected into LLM prompt
ScaleSuitable for thousands of entries; sub-millisecond search up to ~10 k records
def _embed(self, text: str) -> np.ndarray:
    vec   = np.zeros(64)
    words = text.lower().split()
    for wi, word in enumerate(words):
        for ci, ch in enumerate(word):
            vec[(wi * 7 + ci) % 64] += ord(ch) / 1000.0
    norm = np.linalg.norm(vec)
    return (vec / norm) if norm > 0 else vec

# Returns top-3 most similar past decisions as few-shot context
def build_context(self, query: str) -> str:

AutoLearn Module — autolearn.py

Closes the feedback loop between human corrections and future AI behaviour.

accept_correction()

def accept_correction(agent, original_input, original_output, correction):
    # 1. Save corrected mapping to SQLite with approved_by="human", confidence=99
    save_mapping(name, typ, target_code, target_name, 99, approved_by="human")
    # 2. Add to RAG store with human_approved=True metadata
    rag.add(f"{name} {typ}", {**correction, "approved_by": "human"})
    # 3. Log correction to agent_decisions.jsonl for audit
    log_decision("hitl_correction", original_input, correction)

generate_improved_prompt_rules()

After 50+ human corrections, calls the LLM API to synthesise patterns into improved system prompt rules. Result is stored and prepended to future LLM calls.

LLM API Integration

Models Used

RoleModelMax tokensTemperaturePurpose
PrimaryLlama 3.3 70B Versatile4000.1Main reasoning — account mapping, anomaly detection, reconciliation memo
ValidatorLlama 3.1 8B Instant3000.1Dual-validation — provides independent mapping opinion for consensus check
ℹ
Why two models? The primary model (70B) has higher accuracy; the validator (8B) is fast and cheap. When both models agree on a target account, the mapping is auto-approved. Disagreement or low average confidence routes the account to the HITL queue for human review. This eliminates single-model hallucination risk at minimal extra cost.

Retry Strategy

for attempt in range(4):
    try:
        resp = client.chat.completions.create(...)
        break
    except RateLimitError:
        if attempt == 3: raise
        time.sleep(20 * (attempt + 1))  # 20s, 40s, 60s backoff

⚠
Rate limitsThe LLM API provider enforces per-minute request and token limits. The Semaphore(2) limits concurrent calls to stay within these limits. For production loads with more than 50 accounts, consider batching requests with asyncio.sleep(1) between batches or upgrading to a higher-tier API plan.

  1. Add transform function transform_<entity>_record(record) in rules_engine.py
  2. Register in batch_transform()'s dispatch dict
  3. Add to ENTITY_OPTIONS dict in streamlit_app.py
  4. Add column hint to COLUMN_HINTS dict in streamlit_app.py
  5. Optionally add sample data to SAMPLE_DATA dict
