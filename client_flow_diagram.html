AI Accounting Migration System

Reckon Desktop  →  Reckon One  ·  End-to-End Flow

1 · Input Layer
📂 Client Data Upload
  • Source Accounts (COA) — Reckon Desktop Chart of Accounts CSV
  • Target Accounts — Reckon One target account structure
  • Journal Entries — Historical transactions & cutoff date
  • Trial Balance — Source TB vs Migrated TB for reconciliation
2 · Deterministic Rules Engine
⚙️ Rules Engine (rules_engine.py)
Zero AI cost — all deterministic transforms applied first:
  • COA type mapping: 27 Desktop types → Reckon One types (Bank, Equity, Current Assets…)
  • Account code cleaning: max 7 chars, strip date prefixes (AUG-25), dots, special chars
  • Mandatory accounts: Item Sales (Income), Item Purchase (COGS), Rounding (Liability)
  • Transaction mapping: Invoice→Invoice, Cheque→Make Payment, Paycheque→Journal…
  • Tax codes: N→NTD, ADJ-P→AJA, ADJ-S→AJS; start date 01/07/2000
  • Duplicate names: suffix -A, -B; "No Name" → "No Name-C"
  • Multi-currency: multiply amounts × exchange rate (Reckon One has no multi-currency)
3 · Memory & RAG Lookup
🧠 Memory Store + RAG Context
  • Checks SQLite memory for previously approved mappings — reuses instantly
  • LlamaIndex RAG retrieves similar past accounts as few-shot context
  • Learned anomaly patterns from past runs injected into AI prompts
  • Auto-approved accounts skip LLM entirely → faster & cheaper
Needs AI?
4a · AI Mapping (LLM Path)
🤖 Dual-Model LLM Inference (Groq)
Only accounts not resolved by rules/memory reach here:
  • Primary model + Validator model run in parallel
  • Both receive: Reckon rules context + RAG similar accounts + account details
  • If both models agree & confidence ≥ threshold → AUTO APPROVED
  • If confidence medium → REVIEW
  • If models disagree or low confidence → HITL QUEUE
4b · Human-in-the-Loop (HITL)
👤 Human Review Queue
  • Low-confidence or disagreed mappings surfaced to accountant via Streamlit UI
  • Human approves / overrides / provides correct mapping
  • Decision saved to memory → improves future runs automatically
  • If no HITL items, this step is skipped entirely
5 · Anomaly Detection
🔍 Anomaly Agent (Journal Entries)
Scans all journal entries for Reckon-specific issues:
  • Duplicate references, blank refs, special chars in ref numbers
  • Cutoff date violations (revenue recognised in wrong period)
  • GST rounding mismatches, multi-currency not converted
  • Invoice >75 lines (needs consolidation), QTY=0 with amount
  • Bank/CC accounts used as item income/expense accounts
  • Inactive accounts used in transactions during conversion period
Each anomaly rated: HIGH MEDIUM LOW
6 · Trial Balance Reconciliation
📊 Reconcile Agent (CFO-Level)
  • Compares source TB vs migrated TB account-by-account
  • Calculates net variance and flags discrepancies
  • AI explains root cause: GST rounding, mandatory account creation, multi-currency…
  • Returns: PASSED REVIEW FAILED + risk level + next steps
7 · Output & Reporting
✅ Final Report (Streamlit UI / CLI)
  • Account Mapping Table: Source → Target with confidence & status badges
  • Anomaly Report: All issues with severity & recommended actions
  • Reconciliation Summary: TB status, variance breakdown, CFO memo, next steps
  • AutoLearn: Patterns & approved mappings saved to memory for next migration
  • Export: downloadable CSV / JSON results
System Components — Detailed Explanation
📂 Data Sources
The client uploads Reckon Desktop exports: Chart of Accounts CSV, Journal Entries, and Trial Balance sheets. The system also accepts the Reckon One target account structure (pre-configured or uploaded). All data is normalised on ingestion to handle varying column names across different Reckon Desktop versions.
⚙️ Rules Engine
Built from the 45-sheet Reckon mapping spreadsheet (Nov 2024). Covers 30 entity types with deterministic transforms — no AI needed. Handles: 27 COA type mappings, 19 tax codes, transaction type conversions, account code formatting, mandatory account creation, and multi-currency flattening. This runs first and is free (zero LLM cost).
🧠 Memory + RAG
SQLite memory stores every human-approved and auto-approved mapping. On each new run, the system first checks memory — if the same account type/name was seen before, it reuses the result instantly. LlamaIndex RAG retrieves similar past accounts as examples for the LLM. This means the system gets smarter with every migration run.
🤖 Dual-Model AI
Only ambiguous accounts (not resolved by rules or memory) reach the LLM. Two Groq models run in parallel — a primary and a validator. If both agree and confidence is high, the mapping is auto-approved. If they disagree or confidence is low, the account is queued for human review. This dual-check prevents hallucinations from reaching production.
👤 Human-in-the-Loop
Uncertain mappings are surfaced in the Streamlit web UI for the accountant to review. The human can approve, override, or correct each mapping. All decisions are saved to memory — the next migration of a similar client will automatically benefit from these corrections without needing human review again.
🔍 Anomaly Detection
The anomaly agent scans journal entries for 10+ Reckon-specific issue types beyond standard accounting anomalies (duplicates, cutoff issues). It knows about Reckon One's specific constraints: 75-line invoice limit, no multi-currency support, GST rounding rules, restricted account usage. Each finding includes a recommended remediation action.
📊 Reconciliation
The reconcile agent performs a CFO-level trial balance comparison. It calculates account-level variances between the source (pre-migration) and migrated TB, identifies root causes specific to Reckon (mandatory account additions, GST rounding accounts, multi-currency adjustments), and produces a board-ready summary memo with next steps.
✅ AutoLearn & Output
After each run, AutoLearn extracts patterns from anomalies and mappings and saves them to the memory store. Future runs start with this institutional knowledge baked in. The final output is a complete migration package: approved account map, anomaly list with fixes, reconciliation report — all exportable as CSV/JSON for the client's records.
